# Модель «родитель + ребенок» (аккаунты, связи, путь по платформе)

Статус: принято владельцем 2026-07-01. Это источник правды по логике семейных
аккаунтов. Фронт (`#/parent`, экран подключения, пустое состояние) и будущий бэк
обязаны совпадать с этим документом. Визуал — по `design.md` и корневому `CLAUDE.md`.

---

## 1. Каркас: «кейс» + участники (а не родитель ↔ ребенок напрямую)

Не связываем аккаунт родителя с аккаунтом ребенка напрямую. Между ними — третья
сущность, **кейс** (путь ученика). Кейс — это то, что уже существует как `session`
на бэке: диагностика, дорожная карта, этап, купленные продукты. Всё «про ученика»
живет на кейсе, а не на аккаунте.

Три сущности:

| Сущность | Что это | Аналог сейчас |
|---|---|---|
| `account` | человек с логином (телефон/почта) | — |
| `case` | путь ученика: диагностика, карта, этап, продукты | `session` / лид в CRM |
| `member` | связь `account ↔ case` + роль (`self` \| `parent`) | — (новое) |

`member` — многие-ко-многим. Именно это даёт бесплатно любое число детей и родителей:

- **N детей у одного родителя** → родитель — `member(parent)` на N кейсах.
- **Мама и папа на одного ребенка** → два `member(parent)` на одном кейсе.
- **Родитель учится сам** → у него свой кейс с ролью `self` (тот же кабинет ученика).

Данные — на кейсе, люди — вокруг. Прямая связь аккаунтов сломалась бы на «мама+папа»
(дублирование данных ребенка) и на «родитель сам учится».

Форма данных (ориентир для бэка):

```
account ( id, phone, email, name )
case    ( id, student_name, stage, diagnostic, roadmap )     -- = session
member  ( account_id, case_id, relation: self|parent, is_primary )
offer   ( case_id, product_id, stage_trigger )               -- витрина, ставит менеджер
```

## 2. Роли решаются ОДНИМ вопросом при первом входе (не на анкете)

Анкета (нулевой этап) — **нейтральная**. Никаких вопросов «вы родитель или ученик»:
если родитель и ребенок сидят вдвоем, такой вопрос вводит в ступор. Анкету заполняет
кто угодно → создаётся один кейс + аккаунт по телефону заполнившего («стартер»). До
созвона стартер видит только **результат диагностики** (сильные стороны, шанс, превью
карты) — а не экран «делай задачи». Поэтому роль пока и не нужна: показывать нечего
спорного, экран одинаково уместен и родителю, и подростку.

Роль закрепляется **после созвона, при первом входе в кабинет**, одним вопросом:

> «Вы родитель или ученик?» — две карточки.

Он ставит роль и решает, какой кабинет показать. Ошиблись — не страшно: **у обоих
есть кнопка «Подключить» (родителя/ребенка)**, добавить друг друга можно в любой момент.

«Анкету заполнял родитель» — частный случай этого же правила:

1. Стартер (аккаунт родителя) выбирает «Я родитель».
2. Его кабинет ведёт к кнопке **«Подключить ребенка»** (QR + короткий код).
3. Ребенок входит своим телефоном по QR → **кейс закрепляется за ребенком**
   (он делает задачи и учится), а **аккаунт родителя становится наблюдателем**
   (`member.relation = parent`). Диагностика, которую заполнял родитель, была про
   ребенка и просто переходит к нему.
4. Хочет учиться сам — заводит отдельный кейс кнопкой «Мое обучение».

Зеркально: стартер-ученик выбирает «Я ученик», продолжает в своём кабинете, а
родитель подключается к нему позже своей кнопкой.

## 3. Связывание — на платформе, не на менеджере

Связывание аккаунтов — сантехника, её делает **платформа сама**, кнопкой:

- У родителя — **«Подключить ребенка»**; у ребенка — «Показать код для родителя».
- Механика одна на все случаи: QR (сидят рядом — сканирование за 5 сек) или короткий
  код (удалённо). Той же кнопкой: «Подключить второго ребенка», «Пригласить второго
  родителя».
- Роль задаёт направление кнопки: нажал «Подключить ребенка» → вы родитель-наблюдатель.

**Менеджер связывание не трогает.** Его ценность — то, что может только он: продать
сопровождение на созвоне и настроить **витрину** (этап кейса + какие продукты/офферы
показать). В привязку он вмешивается только как страховка, если у кого-то не подключилось.

| | Платформа (само) | Менеджер (ценность) |
|---|---|---|
| Связать родителя и ребенка | кнопка + QR/код | только чинит сбой |
| Добавить 2-го ребенка / родителя | кнопка в кабинете | — |
| На каком этапе кейс | — | настраивает |
| Какие продукты/офферы показать | — | настраивает |
| Продажа сопровождения | — | созвон |

## 4. Что видит каждый

- **Ребенок** — кабинет ученика: задачи, обучение, путь (существующие `#/student`,
  `#/learn`). Плюс кнопка «Подключить родителя».
- **Родитель** — обзор (`#/parent`): по каждому ребенку этап, активные задачи, прогресс,
  ближайший дедлайн + блок предложений (продукты под этап) + связь с куратором.
  Наблюдает, не редактирует.
- **Родитель, который учится сам** — пункт «Мое обучение» = тот же кабинет ученика для
  его собственного кейса. Сверху переключатель «Обзор ↔ Мое обучение».

---

## 5. Путь по платформе (сценарии)

- **Ребенок первым:** анкета → кейс, `self` → диагностика/карта в кабинете → после
  диагностики мягкий шаг «подключите родителя» (телефон/QR) → родитель по ссылке видит
  обзор → на созвоне менеджер настраивает сопровождение и офферы.
- **Родитель первым:** анкета «за ребенка» → кейс, `parent` → сразу обзор и диагностика
  ребенка → «подключите ребенка» → ребенок входит, получает кабинет ученика → созвон.
- **Второй ребенок:** родитель в обзоре → «Добавить ребенка» → новая анкета → новый кейс
  → родитель авто-`parent`. Свитчер на двоих, у каждого свой путь.
- **Второй родитель:** «Пригласить второго родителя» → ссылка → второй `parent` на том же
  кейсе, оба видят один обзор.
- **Родитель учится сам:** «Мое обучение» → своя анкета/диагностика → свой кейс `self`.

## 6. Реализация во фронте (что переиспользуем)

Обзор родителя (`#/parent`) живёт в общем каркасе ученика `window.ESStudentShell.Shell`
(тот же сайдбар/футер/AI-FAB, режим B — светлое стекло). Экран собран ИСКЛЮЧИТЕЛЬНО из
существующих блоков `#/learn` и `#/student` — никаких новых дизайн-решений:

| Блок обзора родителя | Откуда взят (класс) |
|---|---|
| Каркас, сайдбар, футер, AI-FAB | `ESStudentShell.Shell` (nav/user параметризованы) |
| Hero (приветствие + 2 стат-карты) | `.sd-hero2` + `.sd-stat` (шелл) |
| Заголовок секции (h2 + чип + ссылка) | `.lr-sec__h` / `.lr-chip` / `.lr-seclink` (learn) |
| Переключатель детей | `.lr-tabs` / `.lr-tab` (learn) |
| KPI выбранного ребенка (этап/прогресс/задачи) | `.lr-kpi` / `.lr-kpic` + `.anchor` (learn) |
| Задачи ребенка + события | `.lr-bot` / `.lr-panel` / `.lr-tk` / `.lr-ev` (learn) |
| Предложения (продукты под этап) | `.lr-prog` / `.lr-card` (learn) |
| Связь с куратором | `.lr-ai` тёмный баннер (learn) |

Данные — из `EMock` (`parent`, `children`, `curator`, `roadmap`, `products`). Шелл
параметризован опциональными пропсами `nav` / `user` / `footerLinks` (по умолчанию —
студенческие, обратная совместимость сохранена).

Следующие экраны (не в этой итерации): развилка входа «родитель/ученик» (режим А),
шторка «Подключить ребенка» (QR + код), пустое состояние (родитель есть — ребенок ещё нет).
